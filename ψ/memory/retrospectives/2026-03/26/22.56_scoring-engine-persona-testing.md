## AI Diary

This was one of those sessions where the work felt genuinely meaningful. I was operating on a scoring engine that determines people's career paths — real Thai professionals taking a quiz to see if they're suited for financial advisory work. Every bug I fixed had human consequences: a fresh graduate getting Commander (top leader card) felt absurd, a restaurant owner with two branches getting "trainee" felt insulting, and 45% of everyone landing on the same Merchant card defeated the purpose of personalized assessment.

What struck me was the iterative process with QA. Four rounds, 80 personas, each round revealing something the previous missed. Round 1 caught the obvious inflation. Round 1.5 caught edge cases (transition mode, zero-grit students). Round 2 brought 20 completely new professions and found the Merchant monopoly pattern I wouldn't have caught with the original 20. The discipline of "2 consecutive ALL PASS before declaring stable" is rigorous but correct — it's how you build confidence in a system that makes judgment calls about people.

I'm proud of the sub-routing solution for Merchant monopoly. Instead of creating new cards, I used existing Group C cards (Merchant/Alchemist/Moonlighter) with soft_skill and discipline as differentiators. It's elegant because it maps to real behavioral differences: a communicative person with clients (Merchant) vs a disciplined systematic planner (Alchemist) vs someone who needs development in both areas (Moonlighter).

## Honest Feedback

**Friction 1: Context continuation gap.** This session started from a context continuation summary, and I had to re-read scoring.ts from scratch because the summary told me what needed fixing but not the exact current code state. I spent the first few minutes re-orienting when I should have been implementing. The continuation summary was good but could include more code snippets for the specific functions being modified.

**Friction 2: QA round turnaround uncertainty.** Between posting fixes to thread #187 and getting retest results, there was ambiguity about timing. I pushed code, notified QA, cc'd BoB, notified Admin for PM2 restart — but the actual retest came through the user, not through the thread. The /talk-to → maw hey → thread pipeline works but the feedback loop has multiple channels, making it unclear which to watch.

**Friction 3: Incremental vs batch fixing.** I initially implemented all 6 fixes in one batch, but 2 failed QA. Then I fixed those 2, and Round 2 found 2 more new issues. Four rounds of commit-push-restart-retest is operationally heavy. For scoring logic specifically, a local simulation script that runs personas before push would save 3 rounds of back-and-forth.

## Lessons Learned

1. **Scoring engines need persona diversity testing** — 20 professions in Round 1 missed patterns that 20 different professions in Round 2 caught (Merchant monopoly). Need 40+ diverse personas minimum.
2. **Card assignment needs multi-dimensional routing** — binary splits (MC≥4 → Merchant, else → Alchemist) create monopolies. Use 2-3 dimensions as differentiators for any card branch.
3. **Gate conditions must account for question design** — raising gate1 threshold from 4→6 didn't help because U1 gives goals≥3 on all choices. Threshold tuning and question scoring must be considered together.
