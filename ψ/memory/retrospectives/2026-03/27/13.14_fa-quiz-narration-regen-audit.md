## AI Diary

This session taught me something about the limits of code-level verification. I spent significant time doing a thorough audit — checking every question ID against MP3 filenames, every scene.audio against ambient files, every choice against its narration text, even doing git forensics to trace when MP3s were regenerated relative to text changes. Everything checked out perfectly on paper. Every file existed, every mapping was correct, every commit showed MP3s being regenerated after text changes.

But แบงค์ came back with screenshots proving the narration was wrong. The gap was obvious in hindsight: I verified file FORMAT (MPEG headers) and file EXISTENCE, but I cannot listen to audio. The `file` command tells me it's a valid MP3, not that the Thai TTS rendered the correct words. This is the same gap noted in our learnings from yesterday — "file command verifies format but NOT content quality."

The right response was exactly what happened: stop trying to be clever with code-level audits and just regen everything. 48 files, parallel batch, under 5 minutes. The cost of regenerating all MP3s is trivially low compared to the cost of debugging individual mismatches. Nuclear option was the correct option.

The parallel batch approach worked flawlessly again — 4 concurrent curl calls, 12 batches, all 48 files verified. The ElevenLabs API is remarkably stable for Thai TTS with the George voice.

## Honest Feedback

1. **Audit was thorough but missed the point**: I spent 30+ minutes on a meticulous code-level audit proving "no mismatch found" when the user had visual evidence of mismatches. The issue was MP3 content, not code mapping. I should have jumped to "regen all" faster instead of trying to prove the code was correct. The code WAS correct — the audio wasn't. Lesson: when the user says "it's broken" with evidence, trust them and fix it rather than proving your code is right.

2. **Context window pressure from compacted session**: This session inherited from a compacted conversation that covered 7+ tasks (production build, avatar deploy, prologue redesign, QA fixes, scoring filters, sound modal, content audit). Re-reading all the context and figuring out where I left off added friction. The handoff summary was good but I still had to re-read questions.ts fully.

3. **Exit page narration texts weren't cleanly extractable**: The exit page narration content is split across UI text (in ExitPage.tsx) and audio files (no source text stored). I had to compose narration scripts by combining quote + reason + encouragement + challenge from the code. This should be centralized — narration source text should live alongside the audio ID mapping, not scattered across components.

## Lessons Learned

**Nuclear regen beats surgical debugging for TTS audio**: When multiple narration MP3s may be stale, regenerating ALL files (~5 min with parallel batch) is faster and more reliable than identifying which specific files are wrong. Code-level audits verify mapping correctness but CANNOT verify audio content. Always default to full regen when user reports content mismatch.
