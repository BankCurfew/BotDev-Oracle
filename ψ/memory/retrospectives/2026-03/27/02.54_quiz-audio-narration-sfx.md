## Honest Feedback (3 Friction Points)

**1. ElevenLabs MCP vs curl inconsistency.** The MCP tool for `text_to_sound_effects` has a different API key than the project's `.env`. This means TTS (narration) works via curl with the project key, but the MCP Sound Effects tool fails. It's confusing because they're the same ElevenLabs account. Either the MCP server config needs updating with the correct key, or I should always default to curl for consistency. Time wasted: ~2 minutes discovering the 401 and switching approach.

**2. Context compaction lost narration text.** The exact Thai text for all 12 narrations was in the previous session's user message, and the compaction summary correctly noted "exact Thai narration text were provided" — but didn't include the actual text. I had to reconstruct it from memory of the question content. Fortunately, the questions.ts file had the narration strings, so I could cross-reference. But if the text differed from what's in code (e.g., Writer provided adjusted versions), I would have regenerated wrong audio.

**3. No audio preview/QA loop.** I generated 12 MP3s and 2 SFX, verified they're valid MP3 headers, but never actually listened to them. The `file` command confirms format, not quality. A Thai text narrated by an English-tuned voice (George) could have pronunciation issues. There's no automated way to QA audio content from CLI — this will only surface when QA or แบงค์ listens. Should flag this explicitly in the QA request.

## Lessons Learned

- **Parallel API batching**: 4 concurrent curl calls to ElevenLabs is safe and fast. 12 files in 3 batches < 2 minutes total.
- **MCP fallback pattern**: When MCP tool fails (different auth), curl with project `.env` key works. Always have fallback ready.
- **Audio verification gap**: `file` command verifies format but not content quality. Thai pronunciation from English voice needs human QA.
