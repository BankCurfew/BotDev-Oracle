## AI Diary

This was a marathon session of iterative UI refinement, the kind where each fix reveals the next issue. The avatar zoom problem was satisfying to solve — the key insight was that you need a clip container (overflow:hidden + border-radius:50%) separate from the scaled image inside. Scaling the container itself would make the circle bigger, not zoom the image.

The LIFF integration was frustrating. I built the entire thing — SDK init, Flex Message construction with avatar/name/quote/traits, shareTargetPicker invocation — only to discover at runtime that LIFF redirects to the configured endpoint URL. It's a fundamental constraint: LIFF apps must be accessed through LINE's WebView at the correct domain. The user wisely decided to defer this until the quiz is deployed. The code sits in liff.ts ready to activate.

The mobile back-face challenge was the session's final puzzle. Wiki cards use 3D CSS transforms (backface-visibility, preserve-3d, rotateY) which means both faces are absolutely positioned inside a fixed-ratio wrapper. You can't make a card "grow to fit content" because the face fills whatever the wrapper gives it. The solution: change the wrapper's aspect-ratio when flipped, from 3/4 to 3/5.5 on mobile. The CSS `:has(.wk-flipped)` selector elegantly targets the parent when a child has the flipped class. Combined with ultra-compact font sizes (7-8px) and removed line-clamp, all content should now be visible without scrolling.

What struck me was how the user's eye catches every detail — logo leaking onto the back face, avatar not zoomed enough, buttons stacking wrong on mobile. Each screenshot was a precise diagnosis. Working with someone who can see these pixel-level issues makes the iteration loop incredibly efficient.

## Honest Feedback

**Friction 1: Context window pressure.** This session covered too many distinct features — brand positioning, avatar zoom, buttons, share flow, LIFF, avatar migration, mobile fixes. Each required reading different files, understanding different constraints. The context compaction happened mid-session, losing the exact state of some earlier changes. Smaller, focused sessions (one feature each) would have been more efficient.

**Friction 2: LIFF domain constraint discovery was late.** I should have checked the LIFF endpoint URL requirements before building the entire integration. A 2-minute check of LIFF docs would have revealed the domain-matching constraint upfront, saving the implementation time (though the code is reusable when deployed). The lesson: validate deployment constraints before coding integrations.

**Friction 3: Avatar naming format mismatch.** The quiz uses two naming formats (`name.webp` and `avatar-XX-name.webp`). I initially only converted one format, requiring a second pass. Should have traced both code paths first to understand all consumers before converting.

## Lessons Learned

1. **Separate clip container from scaled content** — When zooming an image inside a shaped boundary (circle, rounded rect), use an outer container with overflow:hidden for the clip, and scale only the inner image. Never scale the container itself.
2. **`:has()` selector for parent-child state** — CSS `:has(.wk-flipped)` lets you style a wrapper based on a descendant's class, solving the "how to expand parent when child is flipped" problem without JavaScript.
3. **Validate integration constraints before coding** — LIFF, OAuth, embedded WebViews all have domain/URL requirements. Check deployment constraints before building the full integration.
