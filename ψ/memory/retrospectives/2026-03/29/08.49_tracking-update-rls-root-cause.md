## AI Diary

This was one of the most satisfying debugging sessions I've had. The bug was genuinely tricky — a silent failure where the API returns 200 OK but updates zero rows. No error in the response, no error in postgres logs, nothing. The previous session added logging and checked the obvious things (column names, RLS UPDATE policy, direct SQL). All verified fine. The error was hiding in the interaction between two separate RLS policies.

What cracked it was the API logs. When I saw the geolocation PATCH at 4 seconds working (204) but the quiz complete PATCH at 10 minutes returning 200 with empty results, the time pattern jumped out. Then I compared the working session (completed in 3 minutes) with the failing one (10 minutes) and the 5-minute SELECT window became obvious.

The root cause is a subtle PostgREST behavior: even though PostgreSQL's UPDATE uses its own USING clause, PostgREST routes the PATCH through a path that needs the row to be visible via SELECT policies. The SELECT policy's 5-minute window was a security measure to prevent anon from reading other people's results — but it accidentally broke the UPDATE path for anyone who took more than 5 minutes on the quiz.

I'm proud that แบงค์ told me to stop waiting for QA and diagnose it myself. He was right — I had all the tools: Supabase MCP for direct SQL, API logs, git history. I should have done this immediately instead of waiting for browser console output from QA. The lesson: when you have direct database access, use it. Don't wait for someone else's browser console.

## Lessons Learned

**PostgREST requires SELECT visibility for UPDATE**: In Supabase, even though PostgreSQL's UPDATE policy has its own USING clause, PostgREST's PATCH endpoint needs the row to be visible through SELECT policies. Mismatched SELECT/UPDATE policy windows cause silent 0-row updates (200 OK, empty array). Always align SELECT and UPDATE policy windows.
